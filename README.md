# Лабораторная работа 5
## Сравнение сервисов AWS и Microsoft Azure. Построение кросс-провайдерной сервисной модели

---

## Общая идея и выбранный подход

Исходной точкой для выполнения лабораторной работы 2 послужили результаты лабораторной работы 1, в рамках которой была построена сервисная модель облачных сервисов Amazon Web Services. Эта модель включала иерархию классификации сервисов по уровням:

**IT Tower → Service Family → Service Type → Service Sub Type → Service Usage Type**

Основной задачей второй лабораторной работы стало расширение уже существующей модели на другого провайдера — Microsoft Azure — без изменения её логической структуры. Принципиально важно было не создавать новую классификацию с нуля, а проверить, насколько ранее выбранные принципы разбиения масштабируются на сервисы другого облачного вендора.

Выбранный подход можно сформулировать следующим образом:

> сохранить иерархию и критерии классификации из лабораторной работы 1, но допустить появление новых элементов внутри этой иерархии, обусловленных спецификой Azure.

---

## Подход к сопоставлению сервисов

Работа выполнялась на основе файла биллинга Azure (`Lab_5.xlsx`), содержащего технические поля:
- `Consumed Service`
- `Meter Category`
- `Meter Sub-Category`
- `Meter Name`

Для каждой строки биллинга определялось:
1. какую функциональную роль данный сервис выполняет в облачной архитектуре;
2. к какому уровню иерархии он относится;
3. можно ли сопоставить его с уже существующими значениями из лабораторной работы 1 либо требуется уточнение.

Ключевым источником информации являлась официальная документация Microsoft Azure, а также функциональное сравнение с аналогичными сервисами AWS.

---

## Основные трудности, возникшие в ходе работы

### Различие каталогов сервисов AWS и Azure

AWS и Azure используют разные наименования и уровни детализации сервисов. В Azure присутствуют управляемые сервисы, для которых отсутствуют прямые одноимённые аналоги в AWS, например:
- Azure Data Factory
- Azure API Management
- Azure Machine Learning Studio

Это привело к ключевому вопросу: являются ли значения `Service Type` и нижележащих уровней закрытым словарём или расширяемым классом.

---

### Неоднозначность уровня Service Type

Анализ лабораторной работы 1 показал, что значения `Service Type` уже не являлись чисто абстрактными, а отражали каталожные группы сервисов конкретного провайдера. Следовательно, появление новых значений `Service Type` при переходе к Azure логически допустимо при сохранении верхнеуровневой иерархии (`IT Tower`, `Service Family`).

---

### Баланс между универсальностью и потерей информации

Полный отказ от введения новых обозначений привёл бы к излишнему обобщению и потере информации о характере потребления сервисов Azure. В то же время неконтролируемое добавление новых типов нарушило бы консистентность модели.

В результате был выбран компромиссный подход:
- новые обозначения вводятся только при необходимости;
- каждое нововведение имеет функциональное обоснование;
- структура иерархии не изменяется.

---

## Обоснование введения новых обозначений

### Service Type: Data Factory

**Причина появления:**  
В Azure существует выделенный управляемый сервис оркестрации и трансформации данных — Azure Data Factory, который явно отражён в биллинге.

**Обоснование:**  
Data Factory является управляемым сервисом для ETL и orchestration, не относящимся напрямую к вычислительным ресурсам или хранению данных.

Классификация:
- IT Tower: Cloud Services  
- Service Family: Management Tools  
- Service Type: Data Factory  

---

### Service Type: Machine Learning

**Причина появления:**  
Azure Machine Learning Studio представлен в биллинге как отдельный сервис, тарифицируемый по API-вызовам и операциям обработки.

**Обоснование:**  
Машинное обучение уже рассматривалось в лабораторной работе 1 как отдельное направление (Artificial Intelligence), однако конкретные реализации сервисов у провайдеров различаются.

Классификация:
- IT Tower: Cloud Services  
- Service Family: Artificial Intelligence  
- Service Type: Machine Learning  

---

### Service Sub Type: Processing

**Причина появления:**  
Ряд сервисов (Data Factory, HDInsight, Machine Learning) выполняет обработку данных, не предоставляя инфраструктуру или хранение.

**Обоснование:**  
Подтип `Processing` позволяет логически отделить сервисы вычислительной обработки от сервисов хранения и управления.

---

### Service Usage Type: Requests

**Причина появления:**  
Многие сервисы Azure тарифицируются по количеству вызовов (API calls, activities), а не по времени работы или объёму данных.

**Обоснование:**  
Использование типа потребления `Requests` отражает событийную модель биллинга и имеет прямые аналоги в AWS (например, Lambda invocations).

---

### Service Usage Type: Instance Hour / Bytes / Storage

Эти значения использовались для унификации метрик потребления:
- `Instance Hour` — для вычислительных ресурсов;
- `Bytes` — для передачи и обработки данных;
- `Storage` — для хранения данных.

Они не вводят новой логики, а обеспечивают сопоставимость между провайдерами.

---

## Итоговые выводы

В ходе выполнения лабораторной работы была подтверждена возможность расширения сервисной модели, построенной для одного облачного провайдера, на другого провайдера без изменения принципов классификации.

Все введённые обозначения:
- сохраняют логическую структуру модели;
- отражают реальные функциональные различия сервисов Azure;
- позволяют проводить кросс-провайдерный анализ потребления облачных сервисов.

В результате была сформирована единая кросс-провайдерная сервисная модель, независимая от конкретного вендора и пригодная для аналитического использования.
