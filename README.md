# Лабораторная работа 5
## Сравнение сервисов AWS и Microsoft Azure. Построение кросс-провайдерной сервисной модели

---

## Общая идея и выбранный подход

Исходной точкой для выполнения послужили результаты лабораторной работы 1, в рамках которой мой товарищ построил сервисную модель облаков AWS. Иерархия классификации сервисов по уровням та же:

**IT Tower → Service Family → Service Type → Service Sub Type → Service Usage Type**

Основной задачей второй лабораторной работы стало расширение уже существующей модели на другого провайдера — Microsoft Azure — без изменения её логической структуры, то есть принципиально важно было не создавать новую классификацию с нуля, а проверить, насколько ранее выбранные принципы разбиения масштабируются на сервисы другого облачного вендора.



> в начале работы стало ясно, что хоть и надо сохранить иерархию и критерии с первой лабы, но придется добавить несколько новых элементов внутри этой иерархии, обусловленных спецификой Azure.

---

## Подход к сопоставлению сервисов

Для простоты я отсортировал сначала все по самому верхнему в имеющейся изначально слою `Meter Category`, после чего для каждой строки биллинга определялось:
1. какую функциональную роль данный сервис выполняет в облачной архитектуре;
2. к какому уровню иерархии он относится;
3. можно ли сопоставить его с уже существующими значениями из лабораторной работы 1 либо требуется уточнение.

Ключевым источником информации взял лежащие на поверхности официальная документация Microsoft Azure и функциональное сравнение с аналогичными сервисами AWS, которое лежало, что называется уже под рукой.

---

## Основные трудности, возникшие в ходе работы

### Различие каталогов сервисов AWS и Azure

AWS и Azure используют разные наименования и уровни детализации сервисов. В Azure присутствуют управляемые сервисы, для которых отсутствуют прямые одноимённые аналоги в AWS, например:
- Azure Data Factory
- Azure API Management
- Azure Machine Learning Studio

Это в моменте меня поставило перед вопросом: являются ли значения `Service Type` и нижележащих уровней закрытым словарём или расширяемым классом.

---

### Неоднозначность уровня Service Type

Пристально взглянув на работу моего напарника по первой лабе, я решил, что значения `Service Type` уже не являлись чисто абстрактными, а в сути своей отражали каталожные группы сервисов конкретного провайдера. Стало быть, можно добавить пару новых значений `Service Type` при переходе к Azure, главное - сохранить верхнеуровневую иерархию (`IT Tower`, `Service Family`).

---

### Баланс между универсальностью и потерей информации

Полный отказ от введения новых обозначений привёл бы к излишнему обобщению и потере информации о характере потребления сервисов Azure. В то же время неконтролируемое добавление новых типов нарушило бы консистентность модели. Короче что то, что это, вот в результате и был выбран этакий компромисс:
- новые обозначения ввожу только при необходимости;
- каждому нововведению даю какое-никакое функциональное обоснование;
- ну и структура иерархии не изменяется.

---

## Обоснование введения новых обозначений

### Service Type: Data Factory

**Причина появления:**  
В Azure существует выделенный управляемый сервис оркестрации и трансформации данных — Azure Data Factory, да он и явно отражён в биллинге.

**Обоснование:**  
Data Factory является управляемым сервисом для ETL и orchestration, не относящимся напрямую к вычислительным ресурсам или хранению данных.

Классификация:
- IT Tower: Cloud Services  
- Service Family: Management Tools  
- Service Type: Data Factory  

---

### Service Type: Machine Learning

**Причина появления:**  
Azure Machine Learning Studio представлен в биллинге как отдельный сервис, тарифицируемый по API-вызовам и операциям обработки.

**Обоснование:**  
Машинное обучение уже рассматривалось в лабораторной работе 1 как отдельное направление (Artificial Intelligence), однако конкретные реализации сервисов у провайдеров различаются.

Классификация:
- IT Tower: Cloud Services  
- Service Family: Artificial Intelligence  
- Service Type: Machine Learning  

---

### Service Sub Type: Processing

**Причина появления:**  
Ряд сервисов (Data Factory, HDInsight, Machine Learning) выполняет обработку данных, не предоставляя инфраструктуру или хранение.

**Обоснование:**  
Подтип `Processing` позволяет логически отделить сервисы вычислительной обработки от сервисов хранения и управления.

---

### Service Usage Type: Requests

**Причина появления:**  
Многие сервисы Azure тарифицируются по количеству вызовов (API calls, activities), а не по времени работы или объёму данных.

**Обоснование:**  
Использование типа потребления `Requests` отражает событийную модель биллинга и имеет прямые аналоги в AWS (например, Lambda invocations).

---

### Service Usage Type: Instance Hour / Bytes / Storage

Эти значения взял просто для обобщения метрик потребления:
- `Instance Hour` — для вычислительных ресурсов;
- `Bytes` — для передачи и обработки данных;
- `Storage` — для хранения данных.

Они не вводят новой логики, больше обеспечивают сопоставимость между провайдерами.

---

## Итоговые выводы

В ходе выполнения лабораторной работы была подтверждена возможность расширения сервисной модели, построенной для одного облачного провайдера, на другого провайдера без изменения принципов классификации(ура!)

Все введённые обозначения:
- сохраняют логическую структуру модели;
- отражают реальные функциональные различия сервисов Azure;
- позволяют проводить кросс-провайдерный анализ потребления облачных сервисов.

В результате получили единую кросс-провайдерная сервисную модель, независимую от конкретного вендора, которую можно анализировать ну и творить с ней всякое, что еще заблагорассудится.
